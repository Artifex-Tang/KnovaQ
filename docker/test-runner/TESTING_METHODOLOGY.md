# 测试框架与设计方法论（可推广）

本文把 KnovaQ 这套 E2E 测试框架的**结构**与**设计思路**抽象成通用经验，供其他项目（尤其是含 RAG / 文档解析 / 多服务编排的系统）复用。KnovaQ 专有细节只作举例。

---

## 1. 核心理念：测产品行为，不是测接口返回码

- 不止"接口返回 200 / 有数据"，而要验**端到端业务链路真的成立**。
- RAG 类系统的最小可信链路：`上传 → 解析 → 切块 → 向量化 → 检索 → 生成 → 答案正确`。任何一环断了，前面"绿"都没意义。
- 用例分两层断言：①结构层（解析出块、状态 DONE）②语义层（检索能命中、生成答案语义正确）。

## 2. 套件分层（a→h 模板）

| 层 | 职责 |
|---|---|
| A 功能 | 冒烟 + 主流程，最快，先跑 |
| B 问题域 | 特定业务对象的 CRUD/规则 |
| C 全覆盖 | 页面/接口广度扫描 |
| D 交互 UI | 真浏览器（Playwright）验前端 |
| E 业务逻辑 | 配置组合、参数、多轮、跨语言等深度逻辑 |
| F Bug 回归 | 每个历史 bug 一条用例，防回归 |
| G 管线/压测 | 批量上传→解析→检索全链路 + 负载 |
| H 能力矩阵 | 把"产品支持的每种能力维度"逐一覆盖（见 §4） |

**原则**：先快后慢、先窄后宽。功能套件与压测套件**分开**（压测会污染功能套件，见 §6）。

## 3. 合成测试数据：事实编码 + 双用途 oracle

- 用**生成器**造数据，不依赖人工样本：可控量、可复现、可调格式。
- 把一组**结构化事实**（`question / answer / keywords`）作为单一事实源，编码进**每一种文件格式**。
  - 好处一：同一批问题可跨格式/跨解析器统一问答验证。
  - 好处二：`qa_pairs` 既是 qa 解析器的数据，又是**问答验证的标准答案（oracle）**，一份数据两用。
- 数据量用 env 旋钮控制（如 `H_DOCS_PER_PARSER`、`G_MAX_DOCS_PER_TOPIC`），冒烟用小量、压测设 0=不限。
- **均摊要在"格式匹配"之上**：先按能力维度选对的格式，再在其上均匀取样——别让"按目录顺序填满"导致只覆盖第一种格式（本项目踩过：cap 实现只传了 docx）。

## 4. 能力矩阵覆盖（本项目的 H 套件思路）

产品支持多个"能力维度"（这里是 ragflow 的 15 种 chunk_method 切片方式）时，**逐维度建独立场景**：

1. 按维度要求**选对输入**：格式 + 内容结构都要匹配（qa 要 Q&A 对、table 要带表头表格、tag 要无表头二列、resume 要简历结构、picture 要带文字图片、email 要 eml）。
2. 每维度独立建容器/KB（用该维度的 parser），传匹配文档，解析，验块。
3. 对"可问答"的维度，跑**检索 + 生成 + 语义裁判**；对非问答维度（如 tag/resume）只验结构层。
4. 维度的**前置依赖**先探明：缺模型的维度直接标跳过并写明原因（本项目 audio 无 ASR 模型 → 跳过，而非假装失败）。

## 5. 清理必须走应用级联，绝不裸删存储

- 多存储系统（关系库 + 向量库 + 对象存储 + 队列）里，**直接 DELETE 关系库 = 制造孤儿**：向量库的块、对象存储的文件清不掉，越积越多拖垮引擎。本项目实测攒出 **28209 个 ES 孤儿块 + 73 个 MinIO 孤儿桶**。
- **正解**：调应用自己的删除 API（它级联清所有存储）。各套件 teardown 也走 API 删。
- 兜底清孤儿：向量库 `_delete_by_query match_all`；对象存储站点级删（引擎按需重建）。

## 6. 共享基础设施 = 必须串行 + 抽干

- 所有套件共享同一套后端（单解析器/队列/向量库）。
- **串行跑**，不并发：并发或紧接在压测后跑，会被前一批的解析积压**饿死**；某次 run 的"抽干重启"会掐断另一次 run 的连接（本项目踩过，全 Connection refused）。
- 重负载套件（压测/批量）**跑完抽干**：清队列 + 重启解析服务（只清队列不够，要重启杀掉已吞进内存的任务）。
- 找**吞吐天花板**：阶梯加量测时间与失败率，定一个"时间不太长"的安全上限（本项目 90→55s、300→132s 干净，900→42min 崩 + 并发竞争 bug 复现；安全上限取 ~300）。功能套件用安全量，单独做压测才上大量。

## 7. 配置即测试前置条件

- 跑测试前**核对模型/配置依赖**：嵌入模型、对话模型、各能力维度所需模型（ASR/视觉/rerank）、登录验证码开关等。缺了就先配或先标跳过。
- 区分**运行实例 live 配置** vs **仓库 baked 配置**：live 改的（UI 加的 key）`重置/重建卷会丢且可能无法还原`——记入文档，标注"勿轻易重置"。
- 阻塞类配置举例（本项目）：默认嵌入走本地 CPU 推理 → 卡死，切 API 嵌入；登录验证码开着 → 测试登录全失败，关掉并清缓存；硬编码模型名不存在 → 注册对应 provider。

## 8. 校验 API 契约（字段名！）

- 断言前**读真实 API 返回的字段名**，别想当然。本项目 F 套件读 `chunk_num` 恒为 None（真实字段是 `chunk_count`）→ 假失败。
- 同一信息不同接口字段名可能不同；跨套件保持一致。

## 9. 诊断纪律：可信取证，区分真 bug 与争用

- **在空闲栈复现**：负载下的偶发错误，先抽干到空闲再单独跑，区分"产品缺陷"还是"资源争用/饿死"。本项目 `NoneType.run` 看似 bug，实为 900 文档洪峰触发的连接竞争；空闲单文档解析完美。
- **隔离复现**：单测试单独跑能过 = 套件间争用/污染，不是该用例的问题。
- **别信被污染的观测**：系统代理/TUN 会劫持本机 curl，本机测远程结果不可信 → 用无代理通道或直连。脏 shell 输出会误导（本项目曾误判"后端缺失"）。
- **看引擎日志拿完整 traceback**，定位到行 + 真正的根因调用点，别停在表层异常。

## 10. 报告：每套件 JUnit → 聚合

- 每套件输出 `--junitxml=reports/<suite>.xml`。
- `gen_report.py` 扫描所有 xml → 汇总 Markdown + HTML（总览 + 各套件 + 每用例 通过/失败/跳过/耗时/失败信息）。
- `run_all_tests.sh` 串行跑全部 + 重负载后抽干 + 末尾生成报告，一键复现。

## 11. 容器化 runner

- 测试打成镜像（baked 源码 + 依赖 + 浏览器），与被测栈同网络跑：`docker run --network <net> ... <image> tests/<suite>`。
- 迭代期**挂载** `tests/` 覆盖镜像内源码，改测试免重建镜像；`reports/` 也挂出来取产物。
- 服务名用网络别名（compose service 名），不写死容器名。

---

## 移植到新项目的清单

1. 起 a→h 分层骨架，按产品能力裁剪。
2. 写事实编码的数据生成器（结构化事实源 + 多格式 + oracle 双用）。
3. 列产品能力维度矩阵，逐维度建场景，探明各维度前置依赖。
4. 所有清理走应用 API 级联；写孤儿兜底清理。
5. 测试串行化 + 重负载后抽干；先测吞吐天花板定安全量。
6. 跑测前核对配置/模型依赖；记 live vs baked。
7. 断言前核对真实 API 字段名。
8. 诊断守纪律：空闲复现、隔离复现、可信通道、看引擎日志。
9. JUnit per suite → `gen_report.py` 聚合；`run_all_tests.sh` 一键。
